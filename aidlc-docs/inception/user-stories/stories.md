# ユーザストーリー（User Stories）

`story-generation-plan.md` の回答に基づく方針：
- **分割方法**: ペルソナベース（管理者／一般ユーザ）を基本としつつ、Part 1 は両ペルソナが
  登場する「最小構成のMVPフロー」として時系列で並べる（Q1, Q6）
- **粒度**: INVESTのSmall原則を重視し、可能な限り小さい単位に分割（Q2）
- **受け入れ基準の形式**: チェックリスト形式（Q3）
- **並び順**: Part 1（MVP最小構成の初回セットアップ一式）を最優先し、Part 2（管理者機能拡張）・
  Part 3（一般ユーザ機能拡張）をその後に続ける（Q6）

各ストーリーは `[ID] タイトル` の形式。「関連ペルソナ」欄でペルソナを明示する。

---

## Part 1: MVP最小構成（初回セットアップ〜最初のマスタ参照まで）

### MVP-1: メールアドレスによる登録申請
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、メールアドレスのみを入力して登録申請したい。なぜなら、
パスワード入力の前にまず登録意思を表明したいから。

**受け入れ基準**
- [ ] メールアドレス入力フォームから登録申請を送信できる
- [ ] 既存ユーザか未登録かにかかわらず、常に同じ内容のAPIレスポンスを返す（メールアドレス列挙攻撃対策）
- [ ] 新規メールアドレスの場合のみ、有効期限付きリンクを含む確認メールが送信される

### MVP-2: 確認メールのリンクからパスワードを設定して登録完了
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、確認メールのリンクからパスワードを設定し、登録を完了したい。

**受け入れ基準**
- [ ] リンクのトークンが有効期限内であれば、パスワード設定画面が表示される
- [ ] 有効期限切れトークンの場合はエラーを表示し、再申請を促す
- [ ] パスワード設定完了後、ユーザは「承認待ち」状態になる

### MVP-3: 登録完了ユーザ一覧の確認
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、登録完了（承認待ち）のユーザ一覧を確認したい。

**受け入れ基準**
- [ ] 管理者ダッシュボードに登録完了ユーザの一覧が表示される
- [ ] 一覧にはメールアドレスと登録完了日時が表示される

### MVP-4: ユーザの承認/却下
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、登録完了ユーザを承認または却下したい。

**受け入れ基準**
- [ ] ユーザ一覧から個別に承認/却下を実行できる
- [ ] 承認済みユーザのみログイン可能になる
- [ ] 却下したユーザはログインできない

### MVP-5: 承認/却下結果のメール通知
**関連ペルソナ**: 一般ユーザ、管理者
**ストーリー**: 一般ユーザとして、自分の登録申請が承認または却下されたことをメールで知りたい。

**受け入れ基準**
- [ ] 承認時・却下時それぞれ異なる文面のメールが送信される
- [ ] メール送信はMailPit（開発環境）で確認できる

### MVP-6: ログイン（JWT発行）
**関連ペルソナ**: 一般ユーザ、管理者
**ストーリー**: 承認済みユーザとして、メールアドレスとパスワードでログインし、以降のAPI呼び出しに
使うトークンを取得したい。

**受け入れ基準**
- [ ] 正しい認証情報でログインするとJWTが発行される
- [ ] 誤った認証情報の場合はログインに失敗し、ログイン失敗が監査ログに記録される
- [ ] 未承認ユーザはログインできない
- [ ] トークンには設定可能な有効期限が設定される

### MVP-7: 対象RDBMS接続情報の登録
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、対象RDBMSへの接続情報を登録したい。

**受け入れ基準**
- [ ] 接続情報（種別、ホスト、ポート、DB名、認証情報等）を入力し保存できる
- [ ] 複数の接続を登録できる（`requirements.md` 3.3）
- [ ] 接続設定の登録・変更は監査ログに記録される

### MVP-8: スキーマ取り込み
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、登録した対象RDBMS接続からテーブル/ビュー構造を取り込みたい。

**受け入れ基準**
- [ ] 指定した接続に対してスキーマ取り込み操作を実行できる
- [ ] テーブル/ビューの物理名・コメント・型・制約が内部DBに取り込まれる
- [ ] 取り込んだスキーマ情報は接続ごとに紐づけて保持される
- [ ] 取り込み操作とその結果（成功/失敗）が監査ログに記録される

### MVP-9: テーブル/カラム単位のアクセス権限設定（個別ユーザ）
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、取り込んだスキーマに対し、ユーザ単位でテーブル/カラムの
アクセス権限を設定したい。

**受け入れ基準**
- [ ] テーブルレベルでAllow/Denyを設定できる
- [ ] カラムレベルで「アクセス不可」「読み取りのみ(R)」「読み取り/更新(RU)」「フルアクセス(CRUD、
      全カラム更新可能な場合)」を設定できる
- [ ] 権限設定は「接続 × ユーザ × テーブル/カラム」単位で保持される
- [ ] 権限設定の変更は監査ログに記録される

### MVP-10: アクセス可能なテーブル/ビュー一覧の表示
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、ログイン後に自分がアクセス可能なテーブル/ビューの一覧を見たい。

**受け入れ基準**
- [ ] トップ画面に、選択中の対象RDBMS接続配下でAllow権限のあるテーブル/ビューのみが表示される
- [ ] Deny権限のテーブル/ビューは表示されない

### MVP-11: レコード一覧の閲覧
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、テーブルを選択してレコード一覧をページング付きで閲覧したい。

**受け入れ基準**
- [ ] テーブル選択後、ページング付きでレコード一覧が表示される
- [ ] 読み取り権限（R以上）のあるカラムのみが表示される
- [ ] 大量データ取得（閾値は設定可能、デフォルト100件以上）は監査ログに記録される

## Part 2: 管理者ストーリー（機能拡張）

### ADM-1: ユーザグループの作成
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、複数の一般ユーザをまとめたグループを作成したい。

**受け入れ基準**
- [ ] グループ名を指定してグループを作成できる
- [ ] 既存の一般ユーザをグループに追加/削除できる
- [ ] 1ユーザが複数グループに所属できる

### ADM-2: グループ単位のアクセス権限設定
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、グループ単位でテーブル/カラムのアクセス権限をまとめて設定したい。

**受け入れ基準**
- [ ] グループに対してMVP-9と同じ粒度（テーブルAllow/Deny、カラム4段階）の権限を設定できる
- [ ] グループ権限とユーザ個別権限が両方存在する場合、**ユーザ個別権限が優先**される
      （`story-generation-plan.md` Q5の仮ルール。Application Designで正式決定）
- [ ] グループ権限の変更は監査ログに記録される

### ADM-3: 複数の対象RDBMS接続の管理
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、複数の対象RDBMS接続を登録し、それぞれ独立してユーザ・権限を
管理したい。

**受け入れ基準**
- [ ] 2つ目以降の対象RDBMS接続を追加できる（MVP-7の拡張）
- [ ] 接続ごとに独立したスキーマ・権限設定を持つ
- [ ] ある接続の権限設定が他の接続に影響しない

### ADM-4: アクセス権限設定のYAMLエクスポート
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、設定した権限をYAML形式でエクスポートしたい。

**受け入れ基準**
- [ ] 選択した対象RDBMS接続の権限設定一式をYAMLファイルとしてダウンロードできる
- [ ] エクスポート内容にはユーザ/グループ・テーブル/カラム単位の設定が含まれる

### ADM-5: アクセス権限設定のYAMLインポート
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、YAML形式の権限設定ファイルをインポートし、一括反映したい。

**受け入れ基準**
- [ ] YAMLファイルをアップロードして権限設定に反映できる
- [ ] インポート内容の形式が不正な場合はエラーを表示し反映しない
- [ ] インポート操作は監査ログに記録される

### ADM-6: 監査ログの閲覧・絞り込み
**関連ペルソナ**: 管理者
**ストーリー**: 管理者として、監査ログを閲覧し、必要な条件で絞り込みたい。

**受け入れ基準**
- [ ] 認証イベント・管理操作・データアクセスイベントが一覧表示される
- [ ] 日時・ユーザ・操作種別で絞り込める
- [ ] 一般ユーザは監査ログ画面にアクセスできない

## Part 3: 一般ユーザストーリー（機能拡張）

### GEN-1: レコードの絞り込み・並び替え（UI操作）
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、直感的なUIでレコードの絞り込み条件・ソート順を指定したい。

**受け入れ基準**
- [ ] 読み取り権限以上を持つカラムのみが絞り込み・ソート対象として選択できる
- [ ] 複数条件を組み合わせて絞り込める
- [ ] 絞り込み結果がページング付きで表示される

### GEN-2: WHERE句・ORDER BY句の手入力
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、WHERE句・ORDER BY句を直接SQLで入力して絞り込みたい。

**受け入れ基準**
- [ ] WHERE句・ORDER BY句をテキストで直接入力できる
- [ ] 手入力時はカラムレベルの読み取り権限フィルタが適用されない（設計上の意図的な例外）

### GEN-3: レコードの編集
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、更新権限のあるカラムの値をレコード一覧上で編集したい。

**受け入れ基準**
- [ ] 更新権限（RU以上）のあるカラムのみ編集可能な表示になる
- [ ] 更新権限のないカラムは読み取り専用または非表示になる

### GEN-4: 変更内容の単一トランザクション反映
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、複数レコードの変更をまとめて「反映」ボタンで送信したい。

**受け入れ基準**
- [ ] 「反映」ボタン押下で、変更内容が単一トランザクションとしてバックエンドに送信される
- [ ] 作成・更新・削除のすべてが単一APIエンドポイントで処理される（統一API）
- [ ] トランザクションが一部失敗した場合、全体がロールバックされる
- [ ] データ更新操作は監査ログに記録される

### GEN-5: レコードの作成・削除
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、フルアクセス権限がある場合にレコードを新規作成・削除したい。

**受け入れ基準**
- [ ] フルアクセス（CRUD、全カラム更新可能）権限があるテーブルでのみ作成・削除操作が可能
- [ ] フルアクセス権限がない場合、作成・削除の操作UIが表示されない

### GEN-6: クエリビルダーでのSELECT/FROM/JOIN指定
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、クエリビルダーのFROM/JOINタブでテーブルとエイリアスを指定したい。

**受け入れ基準**
- [ ] FROM/JOINタブでアクセス可能なテーブルを選択できる
- [ ] エイリアスを任意で指定できる
- [ ] 選択したテーブル/エイリアスが他タブのカラム選択肢に反映される

### GEN-7: クエリビルダーでのWHERE/GROUP BY/HAVING/ORDER BY/LIMIT OFFSET指定
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、クエリビルダーの各タブで条件・集計・並び順・件数制限を指定したい。

**受け入れ基準**
- [ ] 各タブで、選択済みテーブル/エイリアスに対しアクセス可能なカラムのみが表示される
- [ ] SELECT/HAVING/ORDER BYタブで集計関数を使用できる
- [ ] LIMIT OFFSETタブで取得件数・オフセットを指定できる

### GEN-8: クエリビルダーからのSQL生成
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、クエリビルダーで指定した内容からSQLを生成したい。

**受け入れ基準**
- [ ] UIでの指定内容から妥当なSQLが生成される
- [ ] 生成したSQLをそのまま保存または実行に連携できる

### GEN-9: 手入力SQLからクエリビルダーへの反映（リバースエンジニアリング）
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、他画面から遷移した際、既存のSQLをクエリビルダーの各タブに
自動反映したい。

**受け入れ基準**
- [ ] クエリ実行画面やクエリ履歴からクエリビルダーへ遷移する際、SQLが解析され各タブに反映される
- [ ] 解析できない複雑なSQLの場合は、その旨を表示する

### GEN-10: クエリの名前付き保存
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、作成したSQLに任意の名前を付けて保存したい。

**受け入れ基準**
- [ ] 手入力またはクエリビルダーで作成したSQLに名前を付けて保存できる
- [ ] 公開（Public）/非公開（Private）を選択できる

### GEN-11: 保存クエリの実行
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、保存済みクエリを選択して実行したい。

**受け入れ基準**
- [ ] Publicなクエリは他ユーザも実行できる
- [ ] Privateなクエリは作成者のみ実行できる
- [ ] 保存クエリ実行時はSQLの編集ができない

### GEN-12: 保存クエリの編集
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、自分が作成した保存クエリを編集したい。

**受け入れ基準**
- [ ] 保存クエリの編集は作成者のみ可能
- [ ] 作成者以外が編集しようとした場合は拒否される

### GEN-13: SQLの直接入力・編集による実行
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、実行画面でSQLを直接入力・編集して実行したい。

**受け入れ基準**
- [ ] 読み取り専用SQLのみ実行できる（更新系SQLは拒否される）
- [ ] `:param` 形式のパラメータが自動検出され、値入力UIが表示される
- [ ] ページングの有無を設定できる

### GEN-14: クエリ実行結果の表示
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、クエリ実行結果を表形式で確認したい。

**受け入れ基準**
- [ ] 実行結果が表形式で表示される
- [ ] 必要に応じてページングされる
- [ ] SQL・パラメータ・結果件数・実行時間・実行日時・実行者が履歴として記録される
- [ ] 保存クエリの場合は実行回数も記録される

### GEN-15: クエリ実行履歴の一覧・絞り込み
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、クエリ実行履歴を一覧・絞り込みして確認したい。

**受け入れ基準**
- [ ] 実行履歴がページング付きで一覧表示される
- [ ] 保存クエリと直接入力クエリが区別して表示される
- [ ] 実行日時（範囲）、実行者（全ユーザ／自分のみ）、SQLテキストで絞り込める

### GEN-16: クエリ履歴からの画面遷移
**関連ペルソナ**: 一般ユーザ
**ストーリー**: 一般ユーザとして、履歴の1件からクエリ実行・クエリ保存・クエリビルダーへ
遷移したい。

**受け入れ基準**
- [ ] 履歴の各行から「再実行」「保存」「ビルダーで編集」の各操作に遷移できる
- [ ] 遷移先の画面には元のSQL・パラメータが引き継がれる
