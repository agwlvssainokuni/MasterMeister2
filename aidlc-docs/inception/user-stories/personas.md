# ペルソナ（Personas）

`story-generation-plan.md` Q4 の回答（簡潔に：役割・主な目的・代表的なゴールのみ）に基づき、
必要最小限の情報でペルソナを定義する。

## ペルソナ1: 管理者（Admin）

- **役割**: システム全体の管理者。ユーザ承認、対象RDBMS接続の設定、スキーマ取り込み、
  アクセス権限設定、監査ログの閲覧を行う。
- **主な目的**: 一般ユーザが安全かつ適切な権限範囲でマスタデータにアクセスできる状態を
  整備・維持すること。
- **代表的なゴール**:
  - 新規ユーザの登録申請を確認し、承認/却下する
  - 対象RDBMSへの接続を登録し、スキーマを取り込む
  - テーブル/カラム単位でユーザ・グループごとのアクセス権限を設定する
  - 監査ログを確認し、不正・異常な操作がないか把握する

## ペルソナ2: 一般ユーザ（General User）

- **役割**: 承認済みアカウントでログインし、許可された範囲でマスタデータの閲覧・編集、
  クエリビルダーやクエリ機能を利用するエンドユーザ。
- **主な目的**: 自分に許可された対象RDBMS接続・テーブル・カラムの範囲内で、マスタデータの
  参照・更新や、任意のデータ抽出（クエリ）を効率よく行うこと。
- **代表的なゴール**:
  - メールアドレスで登録申請し、承認後にログインする
  - アクセス可能なテーブル/ビューのレコードを一覧・絞り込み・編集する
  - クエリビルダーでSQLを組み立て、保存・実行する
  - 過去のクエリ実行履歴を確認し、再実行や再編集につなげる

## 補足: 「ユーザグループ」について

`requirements.md` 3.2 で確定した「一般ユーザのグルーピングによる権限一括設定」は、
独立したペルソナではなく、**一般ユーザに対する管理上の分類・権限付与の単位**として扱う。
ストーリー上は、管理者ペルソナが「グループに対して権限を設定する」操作として現れ、
一般ユーザペルソナ自身はグループの存在を意識せず、結果として付与された権限の範囲内で
操作する（`story-generation-plan.md` Q5: 個別設定がグループ設定より優先されるという
仮ルールを前提とする）。
